Защита от перехвата трафика в локальной сети

Пошаговое пособие по настройке механизмов защиты канального уровня на управляемых коммутаторах (Cisco IOS / Qtech QSW-3750). Для системных администраторов без выделенного бюджета на ИБ.

1 Почему это важно

Локальные сети до сих пор остаются основой инфраструктуры организаций. При всей внешней простоте, значительная часть внутреннего трафика передаётся в открытом виде. Протоколы HTTP, ARP, DHCP не предусматривают шифрования на канальном уровне.

Перехват такого трафика технически несложен: достаточно физического доступа к сетевой розетке и бесплатных инструментов (Wireshark, Ettercap). ARP-spoofing, MAC-flooding, rogue DHCP — реализуются за секунды.

💡 Ключевой принцип пособия

Защита канального уровня — это не вопрос миллионных бюджетов, а результат правильной настройки имеющегося оборудования и чёткой методики, доступной одному системному администратору.

Модель угроз

Все атаки имеют низкую сложность реализации, не требуют дорогого ПО и остаются незаметными для пользователя. Коммутаторы L2 по умолчанию не проверяют легитимность источника кадров.

ARP-spoofing

Отравление ARP-таблиц жертвы и шлюза. Атакующий заявляет, что его MAC соответствует IP шлюза.

Ettercap / Bettercap ~30 сек
MAC-flooding

Переполнение CAM-таблицы коммутатора ложными MAC-адресами. Коммутатор переходит в режим хаба.

macof (dsniff) ~60 сек
Rogue DHCP

Подмена сетевых настроек: ложный шлюз и DNS. Весь трафик жертвы проходит через атакующего.

dhcpd / Yersinia ~10 сек
Пассивный сниффинг

Прослушивание широковещательного трафика в сегменте без активного воздействия на сеть.

Wireshark / tcpdump Мгновенно

Топология сети

Сравнение архитектуры «до» и «после» внедрения комплекса защитных механизмов.

Плоская сеть: один VLAN, один широковещательный домен

Уязвимость: Любой узел видит широковещательный трафик всего сегмента. ARP-spoofing перехватывает трафик между любыми хостами.

Сегментированная сеть: 4 VLAN + комплекс защиты на портах

Защита: VLAN изолируют сегменты. Port Security + DHCP Snooping + DAI + IP Source Guard на каждом порту доступа. ACL на ядре фильтруют меж-VLAN трафик.

VLAN — сегментация сети

Первый и обязательный шаг. Без сегментации злоумышленник видит всю сеть. С VLAN — только свой сегмент. Это сужает поверхность атаки.

План адресации

30Серверный сегмент192.168.30.0/24Коммутатор ядра, порты 20-24

Настройка на коммутаторе доступа

Конфигурация VLAN и портов access 
configure terminal
vlan 10
 name ADMIN
vlan 20
 name STUDENTS
vlan 30
 name SERVERS
vlan 99
 name NATIVE

! Порт доступа для рабочей станции
interface GigabitEthernet 0/2
 switchport mode access
 switchport access vlan 20
 spanning-tree portfast

! Магистральный порт к ядру
interface GigabitEthernet 0/24
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30
 switchport trunk native vlan 99

💡 Почему /24 для всех VLAN?

Унификация масок сокращает ошибки при ручной настройке статических маршрутов и ACL. Номера VLAN кратны десяти для резерва: между 10 и 20 остаётся пространство для VLAN 11–19 (видеонаблюдение, IP-телефония).

Port Security — контроль портов

Решает задачу контроля подключений на уровне порта. Ограничивает количество MAC-адресов и определяет действие при нарушении.

Режимы нарушения

shutdown

Порт административно отключается. Требует ручного включения. Риск отказа в обслуживании.

restrict ← рекомендуется

Кадры отбрасываются, порт остаётся активным, счётчик ошибок растёт. Занятие не прервётся.

protect

Тихое отбрасывание без уведомлений. Не рекомендуется — нарушение остаётся незамеченным.

Конфигурация порта доступа

Port Security + sticky MAC 
interface GigabitEthernet 0/2
 switchport mode access
 switchport access vlan 20
 switchport port-security
 switchport port-security maximum 2
 switchport port-security mac-address sticky
 switchport port-security violation restrict
 spanning-tree portfast

Проверка состояния

Команды диагностики
show port-security
show port-security interface gi0/2
show port-security address

⚠️ Важно: лимит = 2

Один MAC для ПК, второй — для подключаемого IP-телефона или точки доступа. Для ноутбуков с док-станциями (проводной + Wi-Fi) может потребоваться лимит 3.

DHCP Snooping — защита DHCP

Фильтрация DHCP-сообщений на коммутаторе доступа. Строит таблицу привязок (binding database) — фундамент для DAI и IP Source Guard.

Настройка

Глобальная конфигурация и порты 
ip dhcp snooping
ip dhcp snooping vlan 10,20,30

! Порт к легитимному DHCP-серверу (trusted)
interface GigabitEthernet 0/1
 ip dhcp snooping trust

! Пользовательский порт (untrusted) + rate limit
interface GigabitEthernet 0/2
 ip dhcp snooping limit rate 15

Проверка таблицы привязок

Команды проверки
show ip dhcp snooping
show ip dhcp snooping binding

🔗 Критическая зависимость

Без DHCP Snooping не работают DAI и IP Source Guard — им нечем сверяться. Таблица привязок должна сформироваться до включения последующих механизмов.

Dynamic ARP Inspection

Проверяет подлинность ARP-пакетов на основе таблицы привязок DHCP Snooping. Эффективно нейтрализует ARP-spoofing.

Принцип работы

Коммутатор сравнивает IP-адрес и MAC-адрес в ARP-запросе/ответе с записями binding database. При несоответствии — пакет отбрасывается, событие фиксируется в журнале.

Настройка DAI 
ip arp inspection vlan 10,20,30

! Доверенные порты (аплинки к ядру, серверные порты)
interface GigabitEthernet 0/1
 ip arp inspection trust

! Настройка логирования (на реальном Qtech QSW-3750)
ip arp inspection log-buffer logs 256 interval 5

Статические ARP для критичных узлов

Для шлюза и серверов в VLAN 30 — избирательная ручная привязка:

Статические записи на ядре
arp 192.168.30.1 001a.2b3c.4d5e arpa

⚠️ Главная ошибка

Никогда не включайте DAI до DHCP Snooping. Если таблица привязок пуста, коммутатор отбросит ВСЕ ARP-пакеты, включая легитимные. Сеть упадёт через 30 секунд. Сначала DHCP Snooping, дождитесь аренды адресов, потом DAI.

IP Source Guard

Проверяет IP-адрес источника во всех IP-пакетах, передаваемых с порта доступа. Отбрасывает трафик с поддельных IP-адресов.

Настройка на порту доступа 
interface GigabitEthernet 0/2
 ip verify source port-security

Статические привязки для серверов

IP Source Guard опирается на таблицу DHCP Snooping. Хосты со статическими IP необходимо внести вручную:

Статическая привязка
ip source binding 001a.2b3c.4d5e vlan 30 192.168.30.10 interface Gi0/24

📝 Примечание

В сети со смешанной адресацией (DHCP + статика) требуется предварительная инвентаризация. Для пяти серверов — не сложно. Для сотни хостов — составьте белый список заранее.

Порядок внедрения

Порядок не произволен — он продиктован технологическими зависимостями. Нарушение последовательности обречено на неработоспособность защиты или отказ сети.

1

VLAN — сегментация сети

Создание VLAN, назначение портов access/trunk. Не требует других служб, но создаёт границы.

2

Port Security — контроль MAC

Ограничение MAC-адресов на портах доступа. Режим restrict, sticky MAC.

3

DHCP Snooping — таблица привязок

Глобальное включение, настройка trusted-портов к серверам. Ожидание формирования binding database.

⏳ Дождитесь хотя бы одной аренды адресов в каждом VLAN перед следующим шагом!
4

Dynamic ARP Inspection — верификация ARP

Проверка ARP-пакетов по таблице DHCP Snooping. Настройка trusted-портов.

5

IP Source Guard — контроль IP

Проверка IP-адреса источника. Ручные привязки для статических хостов.

6

Организационные меры

Таблица соответствия порт—MAC—рабочее место, аудит конфигураций, физическая охрана шкафов.

Типовые ошибки

Реальные кейсы из практики. Ошибки, которые совершают даже опытные администраторы.

DAI до DHCP Snooping: сеть падает через 30 секунд

Симптом: после включения DAI пропадает связь со всеми хостами в VLAN.

Причина: таблица привязок пуста, DAI нечем сверять ARP-пакеты — отбрасывает все.

Решение: отключить DAI, настроить DHCP Snooping, дождаться аренды адресов, включить DAI снова.

Sticky MAC и смена VLAN: хост не получает адрес

Симптом: после переноса ПК в другую аудиторию (другой VLAN) он не выходит в сеть.

Причина: запись sticky MAC осталась в старой VLAN. Коммутатор видит конфликт.

Решение: очистить привязку вручную: no switchport port-security mac-address sticky <mac>, либо перенастроить порт.

Port Security в shutdown: занятие прервано из-за одного кабеля

Симптом: порт отключился, аудитория без сети.

Причина: режим violation shutdown требует ручного shutdown + no shutdown.

Решение: использовать режим restrict в учебных аудиториях. Настроить SNMP-трапы или syslog для мониторинга.

IP Source Guard блокирует сервер со статическим IP

Симптом: сервер в VLAN 30 недоступен после включения IP Source Guard.

Причина: хост не получал адрес по DHCP — записи в binding database нет.

Решение: добавить статическую привязку командой ip source binding или отключить IP Source Guard на серверных портах.

Потеря таблицы привязок при увольнении администратора

Симптом: новый админ боится включить Port Security, не зная легитимных MAC-адресов.

Причина: отсутствие документации и инвентаризации.

Решение: вести таблицу порт—MAC—рабочее место в любом доступном формате (Excel, wiki, git). Регулярный бэкап конфигураций.
